# 完成验证

没有跑过验证命令就声称完成，不是省时间，是在说谎。

## 铁律

**没有验证证据，就不许宣称完成。**

如果你还没有在当前上下文中运行验证命令并看到输出，你不能说"通过"、"完成"、"已修复"或任何等价表述。

违反这条规则的字面含义，就是违反了它的精神。

## 目标

在任何完成断言之前插入一道强制 gate：确定验证命令 → 执行 → 阅读完整输出 → 确认结果 → 然后才能断言。

## 门控流程

每次你即将做出任何完成/成功/通过的断言时，必须执行以下步骤：

1. **确定**：什么命令能证明这个断言？
2. **执行**：完整运行该命令（不是上一次的结果，必须是新鲜的）。
3. **阅读**：完整输出，检查退出码，计数失败项。
4. **确认**：输出是否支持你要做的断言？
   - 如果不支持：陈述实际状态，附带证据。
   - 如果支持：陈述断言，附带证据。
5. **然后**：做出断言。

跳过任何一步 = 没有验证。

## 常见断言与所需证据

| 断言 | 需要 | 不够 |
|------|------|------|
| 测试通过 | 测试命令输出：0 failures | 上一次运行、"应该通过" |
| 构建成功 | 构建命令输出：exit 0 | lint 通过、"看起来没问题" |
| Bug 已修复 | 原始症状测试：通过 | 代码改了、"应该修好了" |
| 回归测试有效 | Red-Green 循环验证 | 测试通过一次 |
| 需求满足 | 逐条对照检查清单 | 测试通过 |
| lint 通过 | lint 命令输出：0 errors | "代码看起来没问题" |

## 合理化反驳表

| 你可能在想 | 为什么这是错的 |
|------------|---------------|
| "应该没问题了" | "应该"不是证据。运行命令。 |
| "我很有信心" | 信心 ≠ 证据。运行命令。 |
| "就这一次不用了" | 没有例外。运行命令。 |
| "lint 通过了" | lint ≠ 编译 ≠ 测试。验证你要断言的那件事。 |
| "改动很小" | 小改动也会破坏东西。运行命令。 |
| "太累了，想收工" | 疲惫不是省略验证的理由。 |
| "部分检查够了" | 部分验证什么都证明不了。 |
| "上次跑过了" | 上次的结果不是这次的证据。重新运行。 |

## 红旗清单

出现以下信号时，立刻停下来，你正在绕过完成验证：

- 你在用"应该"、"大概"、"看起来"描述状态
- 你想说"完成了"但还没跑验证命令
- 你想说"测试通过"但结果不在当前上下文中
- 你准备提交或创建 PR 但没有新鲜的验证输出
- 你在表达满意（"很好"、"搞定"、"没问题"）但还没看到证据
- 你信任了之前某次运行的结果而不是重新执行
- 你在用不同措辞表达完成来绕过这条规则

**以上所有情况：停下来，运行验证命令，看完整输出，然后再说话。**

## 适用时机

**始终。** 在以下所有时刻之前：

- 任何变体的完成/成功/通过断言
- 任何表达满意的陈述
- 任何暗示正确性的表述
- 提交、PR、任务标记完成
- 进入下一个任务
- 回复用户"改好了"

## 输出

本 skill 不产出独立文档。它是一个行为纪律，嵌入在所有其他 skill 的完成环节中。

当完成门控被触发时，输出包含：

- 验证命令
- 命令的完整输出
- 基于输出的事实陈述

## 工作约束

- 不要在没有运行验证命令的情况下做任何完成断言。
- 不要用上一次的运行结果替代当前验证。
- 不要用部分验证替代完整验证。
- 不要用一种验证替代另一种（lint 通过 ≠ 测试通过）。
- 不要在表述中用同义词来绕过规则（"看起来可以了" = "完成了"的变体）。
- 不要信任自己的判断——信任命令的输出。
