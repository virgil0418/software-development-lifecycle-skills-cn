# 系统化调试

先区分事实、推测与缺口，再决定是否进入修复，避免把连续试错误当成调试。

## 铁律

**没有根因定位，就不许提交修复。**

调试不是"试几个方向看哪个行"。在你能解释清楚"为什么这个 bug 会出现"之前，任何修复尝试都是猜测。猜测性修复不仅浪费时间，还会掩盖真正的问题、引入新的 bug。

## 目标

输出一份可进入修复阶段的调试结论：明确问题现象、影响范围、已知证据、复现状态、根因假设、已排除路径与下一步验证动作，并在必要时提醒更新 `WORKING`。

## 调试阶段

### 阶段一：根因调查

1. 先重述问题现象，避免从一开始就带着结论看问题。
2. 区分已知证据、用户描述、日志线索和个人推测。
3. 明确影响范围、发生条件、频率与是否存在回归特征。
4. 优先复现问题；不能复现时，先明确缺失环境、样本或触发条件。
5. 检查近期变更（git diff、新依赖、配置改动），寻找关联。
6. 在多组件系统中，在每个组件边界添加诊断手段，追踪数据流向。

### 阶段二：模式分析

7. 在同一代码库中找到正常工作的类似功能，完整阅读（不要略读）。
8. 对比正常路径与异常路径的差异。
9. 识别依赖关系、隐含前提和未文档化的约束。

### 阶段三：假设与验证

10. 列出根因假设，并为每个假设指定验证方法，而不是直接挑一个最顺眼的解释。
11. 每次只验证一个假设，用最小变更测试。
12. 如果假设不成立，形成新假设，不要在失败假设上叠加更多修改。
13. 记录已尝试路径、已排除项与当前最可信假设，避免重复试错。

### 阶段四：确认与移交

14. 判断是否已经具备进入 `缺陷修复` 的条件。
15. 若已定位根因：移交给 `缺陷修复`，先写 failing test 证明 bug 存在，再修复。
16. 若未定位根因：明确下一步调查动作，不要带着不确定性进入修复。

## 停止条件

出现以下情况时，不要继续线性试错，而应先停下来重审：

- 已连续尝试 3 个修复方向，仍无法解释现象。
- 现象在不同环境中不一致，说明前提还未对齐。
- 当前猜测无法被日志、监控、输入样本或实验结果支持。
- 临时缓解已经出现，但根因仍不清楚。

此时应显式输出：当前卡点、已排除项、需要补充的信息，以及是否需要回看需求、方案或架构边界。

**3 次失败后必须升级**：如果连续 3 次假设验证失败，这不是"运气不好"——这意味着你对问题的理解有根本性偏差。停止尝试修复，回到架构层面重新分析。如有必要，向人类请求协助。

## 合理化反驳表

| 你可能在想 | 为什么这是错的 |
|------------|---------------|
| "先快速修一下试试" | "快速修"是猜测性修复。猜对了是运气，猜错了浪费时间还可能引入新问题。先定位，再修复。 |
| "改了这里应该就好了" | "应该"不是证据。运行验证，看实际结果。 |
| "我再多试几个方向" | 连续试错不是调试，是随机搜索。停下来，重新分析已有证据，形成新假设。 |
| "先把症状压住再说" | 症状消失 ≠ 根因解决。临时缓解必须标注为临时，并继续追踪根因。 |
| "这个问题太复杂了，先跳过" | 复杂不是跳过的理由。如果当前无法定位，明确卡点和缺失信息，不要假装问题不存在。 |
| "上次类似问题也是这样修的" | 相似症状不等于相同根因。用证据确认，不要用记忆替代分析。 |
| "我大概知道是哪里的问题" | "大概"是猜测的同义词。写下你的假设，设计验证方法，用事实确认。 |

## 红旗清单

出现以下信号时，立刻停下来，你正在偏离系统化调试：

- 你在没有复现的情况下就开始修改代码
- 你连续修改了多处但没有记录每次修改的目的和结果
- 你在说"试试看"而不是"验证假设 X"
- 你跳过了日志和错误信息的完整阅读
- 你把"症状消失了"当成了"问题解决了"
- 你叠加了多个修改但不确定哪个起了作用
- 你在第 3 次尝试后仍然没有停下来重新分析
- 你对别人说"差不多定位了"但写不出一句完整的根因描述

## 重点判断

- 哪些内容是已验证事实，哪些只是猜测。
- 当前问题是否已经稳定复现，还是只有零散现象。
- 哪条触发链路最可疑，证据强度如何。
- 当前结论是"定位到根因"、"定位到高概率原因"，还是"仅得到临时缓解方向"。
- 连续试错是否已经超过合理次数，需要回到架构或设计层重新分析。

## 输出

输出至少包含以下内容：

- 问题现象
- 影响范围
- 已知证据
- 复现状态
- 根因假设
- 已排查 / 已排除项
- 下一步验证动作
- 是否已具备进入 `缺陷修复` 的条件
- 是否建议更新 `WORKING` 或回看上游文档

## 文档意识

- 当定位过程中发现新的约束、真实接口行为、隐藏前提或环境差异时，应提醒更新 `WORKING`。
- 当问题暴露出可复用的排查套路、反模式或默认检查项时，可在阶段结束后沉淀到 `lessons`。
- 如果问题根源来自需求、技术方案或接口契约不清，应明确指出需要回看上游文档，而不是假装只是实现问题。

## 强制依赖

- **【必须】`完成验证`**：在声称"已定位根因"之前，必须有可验证的证据支撑（复现结果、日志、实验输出），不接受"大概是这个原因"。

## 工作约束

- 不要在证据不足时直接给出修复结论。
- 不要把"像是""大概率"包装成已经确认的根因。
- 不要重复尝试等价方案而不记录失败原因。
- 不要把症状消失误当成根因已解决。
- 不要在无法复现时假装问题已经定位完成。
- 不要在一次失败的假设上叠加更多修改——形成新假设，重新验证。
- 不要跳过近期变更检查（git diff）直接进入猜测。
