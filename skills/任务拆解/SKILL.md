# 任务拆解

把大方案拆成小任务，让实现过程可执行、可验证、可追踪。

## 铁律

- **无精确路径不成任务。** 每个任务必须列出涉及的文件精确路径，不接受"相关模块"这类模糊描述。
- **无验证命令不成步骤。** 每个步骤必须附带可直接运行的验证命令和预期输出，不接受"确认功能正常"这类空话。

## 目标

输出一份带有文件结构、依赖关系、验证命令和完成标准的任务清单，并把设计阶段的关键决策映射到执行步骤中。

## 拆解步骤

### 第一步：文件结构前置

在拆任务之前，先列出本次变更的文件地图：

- 哪些文件需要新建，路径和职责是什么
- 哪些文件需要修改，修改哪些部分（标注行范围或函数名）
- 哪些文件是对应的测试文件

设计单元要有清晰边界和明确接口。每个文件一个清晰职责。一起变更的文件放在一起，按职责拆分而非按技术层拆分。在已有代码库中遵循现有模式。

### 第二步：识别闭环与边界

1. 识别最小可交付闭环。
2. 按模块、风险和依赖划分任务边界。
3. 标注每个任务对应的设计决策或目标。
4. 标注前置依赖和建议顺序。
5. 区分本期任务、后续任务与显式延后项。

### 第三步：按原子粒度拆分步骤

每个步骤是一个原子动作。粒度锚定：一个步骤 = 一件事。

典型拆分示例：

- "写失败测试" — 一个步骤
- "运行测试确认失败" — 一个步骤
- "写最小实现" — 一个步骤
- "运行测试确认通过" — 一个步骤
- "提交" — 一个步骤

涉及行为变更的任务，默认按 `测试先行` 的 Red-Green-Refactor 节奏拆分步骤。

### 第四步：补充验证与风险

为每个任务补充：
- 精确的验证命令（可直接复制运行）
- 预期输出（通过/失败/具体值）
- 风险提示与完成标准

## 重点判断

- 哪些任务是第一阶段必须完成的。
- 哪些任务只是优化而非当前闭环必需。
- 哪些验证必须在任务结束时就完成。
- 哪些地方最可能返工，需要提前隔离风险。

## 输出格式

````markdown
## 文件结构

| 操作 | 路径 | 职责 |
|------|------|------|
| 新建 | `src/auth/handler.py` | 处理认证请求 |
| 修改 | `src/api/router.py:45-60` | 添加认证路由 |
| 新建 | `tests/test_auth_handler.py` | 认证处理测试 |

## 任务 1：[任务名称]

**目标**：[一句话说明]
**对应设计决策**：[来自技术方案的哪个决策]
**前置依赖**：无 / 任务 N

**文件**：
- 新建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:行范围`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：写失败测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试确认失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：FAIL — "function not defined"

- [ ] **步骤 3：写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试确认通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```

**风险提示**：[如有]
**延后项**：否
````

## 文档意识

- 当拆解导致范围调整、阶段变化或明显延后项出现时，应提醒更新 `WORKING`。
- 输出应成为后续实施计划的直接输入，而不是只停留在标题列表。

## 工作约束

- 不要输出只有标题没有完成标准的任务。
- 不要把多个独立问题粗暴捆绑在一起。
- 不要把任务拆到失去执行效率。
- 不要遗漏关键依赖关系。
- 不要重新讨论已经在上游确定的产品和技术边界。
- 不要输出没有精确文件路径的任务。
- 不要输出没有可运行验证命令的步骤。
- 不要在涉及行为变更的任务中跳过 Red-Green-Refactor 拆分。
