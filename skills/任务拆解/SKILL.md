# 任务拆解

把大方案拆成小任务，让实现过程可执行、可验证、可追踪。

## 目标

输出一份带有依赖关系、验证方式和完成标准的任务清单，并把设计阶段的关键决策映射到执行步骤中。

## 拆解步骤

1. 识别最小可交付闭环。
2. 按模块、风险和依赖划分任务边界。
3. 标注每个任务对应的设计决策或目标。
4. 标注前置依赖和建议顺序。
5. 为每个任务补充验证方式与完成标准。
6. 区分本期任务、后续任务与显式延后项。

## 重点判断

- 哪些任务是第一阶段必须完成的。
- 哪些任务只是优化而非当前闭环必需。
- 哪些验证必须在任务结束时就完成。
- 哪些地方最可能返工，需要提前隔离风险。

## 输出

输出至少包含以下内容：

- 任务编号
- 任务目标
- 对应设计决策
- 涉及文件或模块
- 前置依赖
- 实施说明
- 验证方式
- 风险提示
- 是否为延后项

## 文档意识

- 当拆解导致范围调整、阶段变化或明显延后项出现时，应提醒更新 `WORKING`。
- 输出应成为后续实施计划的直接输入，而不是只停留在标题列表。

## 工作约束

- 不要输出只有标题没有完成标准的任务。
- 不要把多个独立问题粗暴捆绑在一起。
- 不要把任务拆到失去执行效率。
- 不要遗漏关键依赖关系。
- 不要重新讨论已经在上游确定的产品和技术边界。
