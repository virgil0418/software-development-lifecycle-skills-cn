# 测试先行

先把要证明的行为写成测试，再进入实现，避免"写完再补测试"变成默认习惯。

## 铁律

**没有 failing test，就不许写实现代码。**

这条规则没有例外。不论任务多简单、时间多紧张、你多有信心，都必须先有一个失败的测试证明行为缺口存在，再写最小实现让它通过。

如果你已经写了实现代码却还没有测试：删掉实现，从测试开始。不要把已写的代码"留作参考"，不要"一边看着实现一边补测试"，不要试图"适配"——删除意味着删除。

## 目标

在进入 `编码实施` 之前，先明确本次行为变化、补出最小 failing test、确认失败点正确，再以最小实现通过测试，并为后续 `测试验证` 保留可复用的验证基础。

## 适用条件

以下情况优先使用本 skill：

- 新增功能
- 修复缺陷
- 修改已有行为
- 重构中涉及可观察行为变化

以下情况通常不必强制使用本 skill：

- 纯文档修改
- 纯配置调整且不改变行为
- 纯格式化或重命名
- 一次性临时脚本且不会长期维护

## 测试先行步骤（Red-Green-Refactor）

### RED：写失败测试

1. 先明确本次要改变的行为，不要一上来写一组大而全的测试。
2. 写一个最小测试，只覆盖当前最关键的一个行为。
3. 运行测试，确认它失败在预期点（不是语法错误、不是环境问题、不是测试本身写错）。
4. 失败消息必须能说明"缺了什么行为"，而不是"测试本身有 bug"。

### GREEN：写最小实现

5. 仅为让当前测试通过而写最小实现，不顺手扩功能。
6. 再次运行测试，确认当前行为已通过，并检查是否影响已有测试。

### REFACTOR：小步重构

7. 在保持全部测试通过的前提下做小步重构。
8. 每次重构后立即运行测试确认没有引入回归。
9. 判断是否具备进入 `编码实施` 的条件，或需要下一轮 Red-Green-Refactor。

## 合理化反驳表

| 你可能在想 | 为什么这是错的 |
|------------|---------------|
| "这个改动太简单了，不需要测试" | 简单的改动意味着测试也很简单。写测试的成本几乎为零，但缺少测试的风险不为零。 |
| "我先写完实现再补测试，效果一样" | 先写实现再补测试 = 验证你写的代码做了什么，而不是验证你需要的行为。测试会被实现细节污染。 |
| "我已经手动验证过了" | 手动验证不可复现、不可回归、不可追踪。它是检查，不是验证。 |
| "时间紧，先赶进度" | 没有测试的代码在第一次回归时就会浪费更多时间。"赶进度"恰恰是最需要测试约束的时候。 |
| "删掉已写的实现太浪费了" | 沉没成本不是继续错误流程的理由。带着实现补测试会无意识地让测试适配实现，而不是约束行为。 |
| "这只是个 bugfix，改一行就好" | 一行 bugfix 正是最容易被回归吃掉的改动。先写一个证明 bug 存在的测试，再修复。 |
| "测试框架还没搭好" | 搭测试框架是第一步，不是"以后再说"。没有测试基础设施的项目不适合直接写实现。 |

## 红旗清单

出现以下信号时，立刻停下来，你正在偏离测试先行：

- 你打开了实现文件但还没有写任何测试
- 你在说"先让代码跑起来再加测试"
- 你写的测试在验证实现细节而不是行为
- 你跳过了"运行测试确认失败"这一步
- 你一次写了多个行为的大测试
- 你觉得"这次不需要"
- 你把"测试通过"当成了"功能完成"

## 重点判断

- 当前任务是否真的存在行为变化，值得先补测试。
- 测试是否只验证一个行为，而不是把多个需求混在一起。
- 失败是否发生在预期点，能证明测试确实覆盖到了空缺行为。
- 当前实现是否只是"让测试通过"，还是已经开始无边界扩写。
- 是否已经具备进入 `编码实施` 的条件，还是还需要继续拆更小的测试步长。

## 输出

输出至少包含以下内容：

- 本次行为变化
- 最小测试目标
- 失败验证方式
- 最小实现范围
- 当前测试结果
- 是否建议进入 `编码实施`
- 是否需要后续进入 `测试验证`

## 文档意识

- 如果在测试先行过程中发现需求边界或验收口径不清，应提醒回看需求文档。
- 如果在设计测试时发现接口契约、模块边界或状态流不清，应提醒回看技术方案。
- 如果测试暴露出隐藏前提、历史行为或临时绕过，应在必要时更新 `WORKING`。
- 本 skill 关注实现前后的过程纪律，不替代最终的 `测试验证`。

## 工作约束

- 不要在没有 failing test 的情况下直接进入实现。
- 不要让测试因为语法、环境或夹带多余逻辑而失败。
- 不要一次写多个行为的大测试，导致失败原因不清。
- 不要把"先写完再补测试"当作默认策略。
- 不要把本 skill 产出的过程记录误当成最终验收结论。
- 不要跳过"运行测试确认失败"这一步——这是 RED 阶段的核心验证。
- 不要在 GREEN 阶段写超出当前测试要求的实现代码。
