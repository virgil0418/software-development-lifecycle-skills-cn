# 接收评审

评审反馈需要技术判断，不是情绪表演。验证再实现，该推回就推回。

## 铁律

**不验证，不实现。收到反馈后先确认技术上是否正确，再动手。**

## 目标

对收到的评审反馈做技术评估，按优先级实现合理的修改，推回不合理的建议，确保每次修改都经过验证。

## 响应流程

收到评审反馈时，按以下顺序处理：

1. **阅读**：完整读完所有反馈，不要边读边改。
2. **理解**：用自己的话重述每条反馈的技术要求（或提问澄清）。
3. **验证**：对照代码库确认反馈是否正确。
4. **评估**：对当前代码库来说，这条建议技术上合理吗？
5. **响应**：技术性确认或给出推回理由。
6. **实现**：逐条修改，每改一条运行一次验证。

## 禁止行为

**绝不要说：**
- "你说得对！"（表演性同意）
- "好建议！"、"感谢指出！"（情绪表演）
- "马上改"（在验证之前）

**替代方式：**
- 直接重述技术要求，或直接改了
- "已修复。[简述改了什么]"
- "确认了，是 [具体问题]。已在 [位置] 修复。"
- 用代码证明你听到了反馈，而不是用语言

## 澄清优先

如果反馈中有任何不清楚的条目：

**先停下来，不要实现任何条目。** 先澄清不清楚的部分。

原因：条目之间可能有关联。部分理解 = 错误实现。

```
收到 6 条反馈。
理解 1、2、3、6。不确定 4 和 5。

错误做法：先改 1、2、3、6，再问 4、5。
正确做法：先澄清 4 和 5，全部理解后再开始实现。
```

## 评估反馈

### 来自人类合作者的反馈

- 信任度高，理解后直接实现
- 但如果范围不清，仍然要问
- 不需要表演性同意，直接行动

### 来自外部 reviewer 的反馈

实现之前逐条检查：

1. 对当前代码库来说技术上正确吗？
2. 会不会破坏现有功能？
3. 当前实现这样做是否有原因？
4. 在所有平台/版本上都适用吗？
5. reviewer 是否了解完整上下文？

如果建议有问题 → 推回，附技术理由。

### YAGNI 检查

当 reviewer 建议"应该正式实现"某个功能时：

```
检查代码库中是否真的有调用方。

如果没有调用方 → "这个接口没有被调用。删除它（YAGNI）还是确实有使用场景？"
如果有调用方 → 那就正式实现。
```

## 推回方法

在以下情况推回：

- 建议会破坏现有功能
- reviewer 缺少完整上下文
- 违反 YAGNI（未使用的功能）
- 对当前技术栈来说技术上不正确
- 与人类合作者之前的架构决策冲突

**如何推回：**
- 给出技术理由，不要防御性回应
- 引用代码或测试作为证据
- 提出具体问题
- 如果涉及架构决策，先与人类合作者确认

**如果推回后发现自己错了：**
```
正确做法："确认了，你是对的——[具体原因]。正在修复。"
错误做法：长篇道歉或解释为什么当初会推回。
```

## 实现顺序

收到多条反馈时，按此顺序实现：

1. 先澄清所有不清楚的条目
2. 阻塞性问题（破坏性 bug、安全漏洞）
3. 简单修复（typo、import、格式）
4. 复杂修改（重构、逻辑变更）

**每条修改完成后：**
- 运行验证命令（遵循`完成验证`纪律）
- 确认没有回归
- 再进入下一条

## 合理化反驳表

| 你可能在想 | 为什么这是错的 |
|------------|---------------|
| "reviewer 说了就改吧" | 盲目实现可能引入新问题。先验证。 |
| "先改了再说" | 不理解就改 = 猜测性修改。先澄清。 |
| "这条应该没问题" | "应该"不是验证。对照代码确认。 |
| "不好意思推回" | 技术正确性比社交舒适更重要。 |
| "全改了最快" | 批量修改无法隔离问题。逐条改，逐条验。 |

## 红旗清单

出现以下信号时，你正在偏离正确的 review 接收流程：

- 你在说"好的好的"但还没确认技术上是否正确
- 你在批量修改而不是逐条验证
- 你觉得推回"不太好"所以直接同意了
- 你在实现一条建议但不确定它会不会破坏别的东西
- 你跳过了不确定的条目，先改确定的
- 你在用赞美语言回复 reviewer

## 输出

输出至少包含以下内容：

- 收到的反馈摘要
- 每条反馈的处理决定（采纳 / 推回 / 需澄清）
- 推回项的技术理由
- 已实现项的验证结果
- 是否有未解决的分歧

## 强制依赖

- **【必须】`完成验证`**：每条修改实现后必须运行验证，确认没有回归。

## 工作约束

- 不要在理解之前开始实现。
- 不要批量修改后才运行验证。
- 不要用表演性语言回复 reviewer。
- 不要因为"不好意思"放弃技术正确性。
- 不要在有不确定条目时先改确定的——先全部澄清。
- 不要忽略与已有架构决策的冲突。
