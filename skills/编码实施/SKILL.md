# 编码实施

按当前任务边界实施最小改动，按步骤节奏推进，遇阻即停，每步验证。

## 目标

用尽量小而正确的改动完成当前任务，在实施前后确认相关文档和项目记录是否需要同步更新，并在每个完成断言前提供验证证据。

## 前置检查

开始编码前，确认以下条件已满足：

1. 当前任务有明确的完成标准（来自任务拆解或技术方案）。
2. 涉及行为变更的任务已经通过`测试先行`建立了 failing test。
3. 受影响的文件、模块和配置已识别。
4. 如果任务来自任务清单，清单中有精确文件路径和验证命令。

**如果以上任何条件不满足，停止。** 回到上游 skill（任务拆解或测试先行）补齐前置条件，不要在前提不清的情况下开始写代码。

## 执行节奏

每个任务按以下节拍推进。不要跳步，不要合并多步同时做。

### 步骤 1：重申目标

重述当前任务的目标和完成标准。确认自己知道"做完"是什么样子。

### 步骤 2：最小实现

只写让当前测试通过（或满足当前任务目标）的最小代码。

规则：
- 一次只改一件事
- 不顺手重构无关模块
- 不为了"更优雅"扩大改动面
- 不添加当前任务不要求的功能（YAGNI）

### 步骤 3：运行验证

**【强制】使用`完成验证`纪律。**

运行验证命令，阅读完整输出，确认：
- 当前测试通过
- 已有测试没有回归
- 输出干净（无 error、无 warning）

**验证失败时：** 修代码，不改测试（除非测试本身有 bug）。重新运行直到通过。

### 步骤 4：提交

验证通过后立即提交。提交粒度 = 一个任务或一个有意义的原子变更。

```bash
git add <changed-files>
git commit -m "<type>: <description>"
```

### 步骤 5：检查是否继续

- 当前任务完成？标记完成，进入下一个任务，从步骤 1 开始。
- 发现新问题？记录到任务清单，不要在当前任务中偏离。
- 需要文档同步？记录到 `WORKING`，不要拖到最后。

## 阻塞处理

遇到以下情况时，**立即停止编码**，不要试图绕过：

| 阻塞类型 | 处理方式 |
|----------|----------|
| 任务目标不清 | 回到`任务拆解`或`需求澄清`补齐 |
| 测试约束缺失 | 回到`测试先行`补 failing test |
| 接口或契约不明 | 回到`技术方案`确认 |
| 验证持续失败（3 次以上） | 停止，切到`系统化调试` |
| 改动范围超出当前任务 | 拆出新任务，不要在当前任务中膨胀 |
| 依赖未就绪 | 明确阻塞点，跳到不依赖该项的下一个任务 |

**绝不要做的事：**
- 猜测接口行为然后继续编码
- 忽略测试失败继续推进
- 把"差不多了"当成"完成了"
- 在阻塞状态下写更多代码来"绕过"问题

## 质量 Gate

每个任务完成时，逐条检查：

- [ ] 实现只覆盖当前任务目标，没有范围膨胀
- [ ] 所有验证命令已运行，输出干净
- [ ] 没有引入新的 warning 或 deprecation
- [ ] 改动已提交，commit message 描述了做了什么
- [ ] 如果改动影响行为：有对应测试证明
- [ ] 如果改动影响文档/配置：已同步更新

任何一条不满足？不算完成。

## 输出

输出至少包含以下内容：

- 本次实现目标
- 关键改动点
- 涉及文件或模块
- 验证命令及其输出
- 未覆盖风险
- 是否建议更新 `WORKING` 或上游文档
- 下一步建议

## 强制依赖

- **【必须】`测试先行`**：涉及行为变更的任务，在进入编码实施之前必须已通过测试先行建立测试约束。
- **【必须】`完成验证`**：每次声称步骤完成或任务完成时，必须遵循完成验证的门控流程。

## 文档意识

- 如果改动影响产品行为或范围，应提醒回看需求文档。
- 如果改动影响实现边界、接口或架构，应提醒回看技术方案。
- 如果出现实测结论、阻塞、临时 workaround、返工原因，应更新 `WORKING`。
- 如果本次实现暴露出可复用模式或反模式，可在阶段结束后触发 `lessons`。

## 工作约束

- 不要在前置条件不满足时开始编码。
- 不要在当前任务外顺手重构无关模块。
- 不要为了"更优雅"扩大改动面。
- 不要在应先补测试的行为改动上直接绕过`测试先行`。
- 不要跳过验证直接宣称完成。
- 不要在验证失败时修改测试来让它通过。
- 不要在阻塞状态下继续堆代码。
- 文档、配置、测试如需同步，要一起收口。
