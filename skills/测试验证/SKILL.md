# 测试验证

先定义怎么证明成功，再汇报是否完成；没有证据时，应先说明缺口，而不是提前宣布通过。

## 目标

让“完成”有清晰的验收标准和验证证据支撑，并确保接口行为与既定契约一致，同时显式说明验证缺口、未覆盖风险与当前结论的可信边界。

## 与其他 skill 的边界

- 本 skill 不替代 `测试先行`；`测试先行` 负责实现过程中的测试纪律，本 skill 负责交付前后的验证闭环。
- 本 skill 不替代 `系统化调试`；若根因未明、现象不稳或复现条件缺失，应先回到调试阶段。
- 本 skill 不替代 `编码实施`；它负责证明结果，不负责扩写实现。
- 本 skill 可以复用 `测试先行` 产出的测试基础，但不能把“已有测试”直接等同于“验证充分”。

## 验证步骤

1. 明确验收标准，先回答“什么结果算通过，什么结果不能接受”。
2. 明确测试范围，覆盖正常路径、边界路径和异常路径。
3. 识别验证方式：单元测试、接口测试、集成测试、回归测试、人工验证或实测观察。
4. 对照 API 契约或模块契约验证请求、响应、错误语义、状态变化和兼容性约束。
5. 记录结果时区分：已验证结论、待验证缺口、暂未覆盖风险。
6. 判断当前证据是否足以进入下一阶段；若不足，应说明阻塞点而不是给出过度肯定结论。

## API 契约要求

- 只要存在 API、函数边界、模块契约或服务交互，就应验证输入输出是否符合约定。
- API 测试至少要覆盖：正常请求、非法输入、边界输入、错误响应、兼容性风险、状态一致性。
- 如果技术方案里给出了请求/响应示例、字段表或错误码说明，验证应显式对照这些契约。
- 若契约本身不明确，应先指出验证缺口，而不是假装测试通过。
- 对外部 API、第三方服务或不稳定协议，优先通过实测确认真实行为，再决定测试范围与结论强度。

## 重点判断

- 当前结论是“已验证通过”，还是“暂时无证据表明失败”。
- 正常路径、边界路径、异常路径是否都被覆盖到合理程度。
- 契约验证是否真的对照了字段、错误语义和状态变化，而不是只看“代码能跑”。
- 现有测试结果是否足以支持交付，还是仍存在关键验证缺口。
- 当前未覆盖风险是否会影响上线、合并或进入下一阶段。

## 输出

输出至少包含以下内容：

- 验收标准
- 测试范围
- 测试方法
- API / 模块契约验证结论
- 测试结果
- 验证缺口
- 未覆盖风险
- 结论

## 常见错误

- 把“代码能跑”当成验证完成。
- 只覆盖正常路径，不覆盖边界和异常路径。
- 没有真实执行验证，却写“应该没问题”或“理论上通过”。
- 契约未定义清楚，却直接给出肯定结论。
- 把已有测试文件存在，误当成当前场景已经验证充分。

## 强制依赖

- **【必须】`完成验证`**：在断言"验证通过"或"测试覆盖充分"之前，必须运行验证命令并提供证据。不接受"应该没问题"或"理论上通过"。

## 工作约束

- 不要把“代码能跑”当作验证完成。
- 不要只覆盖正常路径。
- 不要隐藏失败结果或跳过复现步骤。
- 如果验证条件不足，要明确写出缺口。
- 不要在 API 契约未定义时给出过度肯定的测试结论。
- 不要把“尚未发现问题”包装成“已经证明正确”。
