# 测试验证

先定义怎么证明成功，再汇报是否完成。

## 目标

让“完成”有清晰的验收标准和验证证据支撑，并确保接口行为与既定契约一致。

## 验证步骤

1. 明确验收标准。
2. 覆盖正常路径、边界路径和异常路径。
3. 识别是否需要接口测试、集成测试、回归测试。
4. 对照 API 契约验证请求、响应、错误语义和状态变化。
5. 记录结果与剩余风险。
6. 判断是否允许进入下一阶段。

## API 测试要求

- 只要存在 API、函数边界、模块契约或服务交互，就应验证输入输出是否符合约定。
- API 测试至少要覆盖：正常请求、非法输入、边界输入、错误响应、兼容性风险、状态一致性。
- 如果技术方案里给出了请求/响应示例、字段表或错误码说明，测试应显式对照这些契约。
- 若契约本身不明确，应先指出测试缺口，而不是假装测试通过。
- 对外部 API、第三方服务或不稳定协议，优先通过实测确认真实行为，再决定测试范围。

## 输出

输出至少包含以下内容：

- 验收标准
- 测试范围
- 测试方法
- API 契约验证结论
- 测试结果
- 未覆盖风险
- 结论

## 工作约束

- 不要把“代码能跑”当作验证完成。
- 不要只覆盖正常路径。
- 不要隐藏失败结果或跳过复现步骤。
- 如果验证条件不足，要明确写出缺口。
- 不要在 API 契约未定义时给出过度肯定的测试结论。
