# 代码评审

按风险级别找问题，先处理阻塞项，再讨论优化项。

## 目标

评估实现是否满足需求和方案，是否存在会影响正确性、稳定性和维护性的关键问题，并识别哪些问题值得沉淀为长期经验。

## 评审步骤

1. 对照需求、方案和任务目标检查实现。
2. 检查遗漏分支、异常处理和边界条件。
3. 检查复杂度是否不必要地升高。
4. 检查是否缺少必要测试或验证。
5. 判断问题是一次性缺陷，还是模式性问题。
6. 对模式性问题考虑触发 `lessons`。

## 输出

输出至少包含以下内容：

- 总体结论
- 严重问题
- 重要问题
- 建议优化
- 可接受风险
- 是否建议沉淀 `lessons`

## 工作约束

- 先找阻塞合入的问题，再谈风格建议。
- 问题描述要具体到行为、条件和影响。
- 能给修复方向时尽量给修复方向。
- 不要用空泛评价替代问题定位。
