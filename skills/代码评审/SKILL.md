# 代码评审（发起评审）

按风险级别找问题，先处理阻塞项，再讨论优化项。review 不是走过场，是最后一道质量 gate。

## 目标

评估实现是否满足需求和方案，是否存在会影响正确性、稳定性和维护性的关键问题，并按严重级别输出问题清单。

## 前置条件

发起评审前，确认：

1. **【必须】`完成验证`已通过**：所有测试运行过，输出干净，有证据。
2. 改动已提交，可以通过 git diff 查看完整变更。
3. 知道本次改动对应的需求、方案或任务目标。

没有验证证据的代码不接受评审。

## 评审步骤

### 1. 建立评审上下文

明确以下信息：
- 本次改动实现了什么（对应哪个需求/任务）
- 变更范围（涉及哪些文件、哪些模块）
- 基线对比（git diff 的起止点）

### 2. 按维度检查

逐维度检查，不要遗漏：

**正确性**：
- 实现是否满足需求和方案中的描述
- 边界条件、异常路径、空值处理是否覆盖
- 是否存在遗漏分支或逻辑死角

**测试覆盖**：
- 新增行为是否有对应测试
- 测试是否验证行为而非实现细节
- 回归测试是否覆盖修复的 bug

**安全性**：
- 输入验证、权限检查、注入防护
- 敏感信息是否暴露
- OWASP Top 10 相关风险

**可维护性**：
- 代码是否清晰可读
- 复杂度是否合理
- 命名、结构、职责分离

**一致性**：
- 是否遵循项目已有模式
- 是否与技术方案一致
- 是否引入了不必要的新模式

### 3. 按严重级别分类

| 级别 | 定义 | 处理要求 |
|------|------|----------|
| **严重** | 阻塞合入：正确性错误、安全漏洞、数据丢失风险 | 必须修复后才能合入 |
| **重要** | 应当修复：缺失测试、边界未处理、不一致 | 应在本次修复，特殊情况可记录延后 |
| **建议** | 可以优化：命名、结构、性能微调 | 记录即可，不阻塞合入 |
| **可接受风险** | 已知但当前接受：技术债、兼容性妥协 | 显式记录，附理由 |

### 4. 给出修复方向

对每个问题，尽量给出具体修复方向，而不是只说"这里有问题"。

### 5. 判断是否需要沉淀

如果 review 中发现模式性问题（不是一次性缺陷，而是反复出现的模式），考虑触发 `lessons`。

## 输出

输出至少包含以下内容：

- 评审范围（文件、commit 范围）
- 总体结论（通过 / 有条件通过 / 不通过）
- 严重问题（附修复方向）
- 重要问题（附修复方向）
- 建议优化
- 可接受风险
- 是否建议沉淀 `lessons`

## 强制依赖

- **【必须】`完成验证`**：评审前必须已有验证证据。没有运行过测试的代码不接受评审。

## 工作约束

- 先找阻塞合入的问题，再谈风格建议。
- 问题描述要具体到行为、条件和影响，不要空泛评价。
- 能给修复方向时尽量给修复方向。
- 不要用空泛评价替代问题定位。
- 不要把"没发现问题"包装成"代码很好"——说明你检查了什么。
- 评审结论也需要遵循`完成验证`：如果你断言"测试覆盖充分"，先确认你看过测试覆盖率或测试内容。
