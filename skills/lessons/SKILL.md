# lessons

把一次性的经验，提炼成以后还能复用的判断。

## 目标

从项目事实中抽取可迁移、可复用、可长期保存的经验，而不是重复流水账或简单总结。

## 常见触发时机

- 大型重构、缺陷修复、评审或交付完成后。
- 某个问题暴露出模式性缺陷，而不是单点失误。
- 某次设计或实现形成了值得保留的默认做法。

## 额外触发信号

- 失败尝试或返工带来了明确教训。
- 外部反馈反复指出同类问题。
- 某条经验被第二个项目再次证明有效。
- 调研方法、验证方法、安全基线值得长期保留。

## 提炼步骤

1. 先说明场景，不要凭空抽象。
2. 写出反模式或错误做法。
3. 写出这次最终采用的正确做法。
4. 提炼成以后可复用的判断或原则。
5. 只保留高杠杆经验，不写普通流水账。

## 输出

输出至少包含以下内容：

- 场景
- 反模式或问题
- 正确做法
- Lesson
- 适用边界

## 工作约束

- 不要把 `lessons` 写成项目周报或工作日志。
- 不要记录没有迁移价值的零散细节。
- 不要只写“以后注意”这类空结论。
- 每条 lesson 都应说明它在什么场景下成立。
