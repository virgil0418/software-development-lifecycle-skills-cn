# 生命周期总控

先判断阶段，再切换到合适的下游 skill，不要在上下文不足时直接进入编码。

## 目标

把用户当前请求映射到正确的软件开发阶段，并给出下一步最合适的 skill。

## 执行流程

1. 识别任务类型：新增功能、重构、优化、修复、交付。
2. 判断信息是否足以进入当前阶段。
3. 如果缺少关键信息，指出最小缺口。
4. 选择下一个 skill，而不是自己无限扩张。
5. 在每次阶段切换前说明切换依据。

## 阶段判断规则

- 目标、边界、验收不清时，切到“需求澄清”。
- 需求明确但实现路径未定时，切到“技术方案”。
- 方案确定但任务过大或执行顺序不清时，切到“任务拆解”。
- 当前目标已明确且可以动手改代码时，切到“编码实施”。
- 需要证明结果、补测试、确认回归时，切到“测试验证”。
- 需要把关质量、按严重级别输出问题时，切到“代码评审”。
- 出现异常、bug、回归、线上告警时，切到“缺陷修复”。
- 准备合并、交付、总结、说明风险时，切到“交付收尾”。

## 输出

输出至少包含以下内容：

- 当前阶段
- 判断依据
- 推荐下游 skill
- 进入下一阶段前仍缺少的信息
- 当前不建议直接做的事情

## 工作约束

- 不要在需求不清时直接给实现细节。
- 不要在方案未定时直接大规模拆任务。
- 不要把“用户很急”当作跳过阶段的理由。
- 不要在没有验证依据时宣称任务已完成。
