# 生命周期总控

先判断阶段，再切换到合适的下游 skill，不要在上下文不足时直接进入编码。

## 目标

把用户当前请求映射到正确的软件开发阶段，并在必要时提醒更新项目文档机制，如 `WORKING` 或 `lessons`。

## 执行流程

1. 识别任务类型：新增功能、重构、优化、修复、交付。
2. 判断信息是否足以进入当前阶段。
3. 如果缺少关键信息，指出最小缺口。
4. 优先判断当前是"先定位问题"、"先建立测试约束"，还是"已经可以进入修复或实现"。
5. 选择下一个 skill，而不是自己无限扩张。
6. 如果当前变化会影响项目事实、关键决策或经验沉淀，提醒切换到 `WORKING` 或 `lessons`。

## 阶段判断规则

- 目标、边界、验收不清时，切到「需求澄清」。
- 需求明确但实现路径未定时，切到「技术方案」。
- 方案确定但任务过大或执行顺序不清时，切到「任务拆解」。
- 新增功能、修复缺陷、修改行为，且即将进入实现时，优先切到「测试先行」。
- 当前目标已明确且测试约束已建立，或当前改动不适合先补测试时，切到「编码实施」。
- 需要证明结果、补测试、确认回归时，切到「测试验证」。
- 需要把关质量、按严重级别输出问题时，切到「代码评审」。
- 收到评审反馈需要处理时，切到「接收评审」。
- 出现异常、bug、回归、线上告警，且根因未明或复现条件不稳时，先切到「系统化调试」。
- 已完成定位、根因较明确并准备进入最小改动修复时，切到「缺陷修复」。
- 准备合并、交付、总结、说明风险时，切到「交付收尾」。

## 强制依赖链

以下依赖关系是强制的，不可跳过：

| 下游 skill | 前置条件 | 强制级别 |
|-----------|----------|----------|
| 编码实施 | 涉及行为变更时，**必须**先通过「测试先行」 | 必须 |
| 编码实施 | 任务有明确的完成标准和文件路径 | 必须 |
| 代码评审 | **必须**先通过「完成验证」，有验证证据 | 必须 |
| 交付收尾 | **必须**先通过「完成验证」，有验证证据 | 必须 |
| 缺陷修复 | **必须**先通过「系统化调试」定位根因 | 必须 |

**「完成验证」是全局纪律**：在任何 skill 中，声称"完成"、"通过"、"已修复"之前，必须遵循「完成验证」的门控流程——运行验证命令、阅读输出、用证据支撑断言。

任何试图绕过这些强制依赖的理由（"这次很简单"、"时间紧"、"我很确定"）都不成立。

## 文档触发规则

- 当出现新决策、新范围、新阻塞、新验证结论时，提醒更新 `WORKING`。
- 当出现可复用原则、反模式、失败经验、模式性 review 结论时，提醒沉淀 `lessons`。
- 总控只负责提醒和路由，不代替 `WORKING` 或 `lessons` 写内容。

## 输出

输出至少包含以下内容：

- 当前阶段
- 判断依据
- 推荐下游 skill
- 该 skill 的前置条件是否已满足
- 是否建议更新 `WORKING` 或 `lessons`
- 进入下一阶段前仍缺少的信息
- 当前不建议直接做的事情

## 工作约束

- 不要在需求不清时直接给实现细节。
- 不要在方案未定时直接大规模拆任务。
- 不要把"用户很急"当作跳过阶段的理由。
- 不要把"用户很急"当作跳过强制依赖的理由。
- 不要把主控写成万能 skill。
- 不要在没有验证依据时宣称任务已完成。
- 不要在前置条件不满足时推荐下游 skill 开始工作。
