# 技能编写

用 TDD 的方法写 skill：先证明 agent 会犯错，再写 skill 堵住漏洞，最后验证 skill 确实有效。

## 铁律

**没有失败场景，就不许写 skill。**

不要从"我觉得应该有这个 skill"开始，而要从"agent 在这个场景下确实会犯错"开始。先构造一个压力场景证明问题存在，再写 skill 解决它。

## 目标

输出一个符合仓库规范、经过压力测试验证、能有效约束 agent 行为的 SKILL.md 文件。

## 核心概念：TDD 应用于文档

| TDD 概念 | Skill 编写对应 |
|-----------|---------------|
| 测试用例 | 压力场景（agent 容易犯错的情境） |
| 生产代码 | SKILL.md 文档 |
| 测试失败（RED） | agent 在没有 skill 时偏离预期 |
| 测试通过（GREEN） | agent 在有 skill 时遵守约束 |
| 重构（REFACTOR） | 收紧漏洞、补充反驳，同时保持通过 |

## Skill 类型

| 类型 | 特征 | 示例 |
|------|------|------|
| 纪律型 | 强制执行特定流程，阻止常见偏离 | 测试先行、系统化调试 |
| 方法型 | 提供具体步骤和操作指南 | 任务拆解、需求澄清 |
| 记录型 | 约束信息记录的格式和触发时机 | WORKING、lessons |

## 编写步骤

### RED 阶段：构造失败场景

1. **识别问题**：agent 在什么场景下会犯什么错？从实际使用中观察，不要凭空想象。
2. **构造压力场景**：设计一个具体情境，包含时间压力、沉没成本或复杂性，让 agent 有动机偏离正确做法。
3. **记录基线行为**：在没有 skill 的情况下，让 agent 面对该场景，逐字记录它的偏离行为和合理化借口。
4. **提取合理化模式**：从基线行为中提取 agent 用来绕过正确做法的典型借口。

压力场景设计要点：
- 场景必须具体，不能是抽象描述
- 必须包含让 agent 想走捷径的诱因（时间紧、看起来简单、已有代码……）
- 目标是暴露 agent 会怎么绕过规则，而不是测试 agent 的能力

### GREEN 阶段：写最小 skill

5. **创建 SKILL.md**：针对 RED 阶段发现的具体偏离行为写 skill，而不是写一个通用指南。
6. **遵循结构规范**（见下文"文件结构"）。
7. **针对每个合理化借口写反驳**：用合理化反驳表逐条回应 RED 阶段提取的借口。
8. **验证约束力**：让 agent 在有 skill 的情况下重新面对同一压力场景，确认行为改善。

### REFACTOR 阶段：收紧漏洞

9. **寻找新的绕过方式**：agent 可能会用新的借口或变体来绕过 skill。
10. **补充红旗清单**：把新发现的偏离信号加入红旗清单。
11. **补充约束条款**：在工作约束中堵住新发现的漏洞。
12. **重新验证**：确认收紧后 skill 仍然有效且不会过度约束正常操作。

## 文件结构

```
skills/
  skill-name/
    SKILL.md              # 主文件（必须）
    supporting-file.*     # 仅在必要时添加
```

### SKILL.md 结构规范

```yaml
---
name: skill-name
description: 用于……的场景。适用于……
---
```

**frontmatter 规则**：
- `name`：只用中文或英文，不含特殊字符
- `description`：以"用于"开头，描述触发条件和适用场景，不要概括工作流

**必须包含的章节**：
- `# 标题`
- `## 目标`
- `## 输出`
- `## 工作约束`

**纪律型 skill 还应包含**：
- `## 铁律` — 不可违反的核心规则
- `## 合理化反驳表` — 借口与反驳的对照表
- `## 红旗清单` — 偏离信号的自检列表

### description 编写规则

description 是 agent 决定是否加载 skill 的唯一线索。写好 description 直接决定 skill 是否会被使用。

- 描述触发条件，不要概括工作流
- 用具体的场景、症状和关键词
- 不要写成 skill 的内容摘要（agent 会用摘要当快捷方式跳过正文）

好的示例：
> 用于在 bug、异常、回归、flaky 问题和根因不明的故障场景中先做系统化定位。

坏的示例：
> 系统化调试技能，包含根因调查、模式分析、假设验证等步骤。

## 约束力强度指南

不同措辞的约束力不同，根据需要选择：

| 强度 | 措辞 | 适用场景 |
|------|------|----------|
| 铁律 | "不许""禁止""必须" | 核心流程纪律，违反会导致严重后果 |
| 强约束 | "不要""不接受" | 重要规则，偏离会降低产出质量 |
| 建议 | "优先""应""推荐" | 最佳实践，可根据具体情况调整 |

纪律型 skill 的核心规则必须用铁律强度。方法型 skill 主要用强约束和建议。

## 与现有验证体系对接

新 skill 创建后，应同步更新：

1. **行为契约测试**（`tests/skill-behavior/cases.json`）：为新 skill 添加 `must_contain` 和 `must_not_contain` 检查项。
2. **结构校验**（`scripts/validate_skills.py`）：确保新 skill 通过结构校验。
3. **能力地图**（`docs/能力地图.md`）：补充新 skill 的职责、适用时机、上下游关系。

## 输出

- 符合结构规范的 SKILL.md
- 压力场景记录（至少包含：场景描述、基线偏离行为、合理化借口提取）
- 行为契约测试用例（添加到 `tests/skill-behavior/cases.json`）

## 文档意识

- 新建 skill 应同步更新 `能力地图.md`，明确上下游关系。
- 如果新 skill 与已有 skill 存在边界重叠，应在两个 skill 的"不负责"中相互引用。
- 重大的 skill 设计决策应记录到 `WORKING`。

## 工作约束

- 不要在没有压力场景的情况下凭空写 skill。
- 不要写只有标题和章节但没有约束力的 skill。
- 不要让 description 变成工作流摘要。
- 不要跳过基线测试直接写 skill。
- 不要忽略合理化借口——它们是 skill 最需要堵住的漏洞。
- 不要创建与已有 skill 职责重叠的新 skill，优先考虑加强已有 skill。
