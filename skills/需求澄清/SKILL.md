# 需求澄清

先把“到底要做什么”讲清楚，再进入设计或编码。

## 目标

输出一份可讨论、可设计、可验收的需求说明，而不是停留在含糊表述上。

## 澄清步骤

1. 重述用户原始诉求，确认你理解的是不是同一件事。
2. 追问业务目标，而不只盯着表面功能。
3. 识别范围内事项与范围外事项。
4. 找出会影响方案选择的关键约束。
5. 明确验收标准与失败判定。
6. 把未确认问题单列出来。

## 必问维度

- 背景是什么
- 目标是什么
- 谁是使用者或受影响方
- 范围内包含什么
- 范围外排除什么
- 有哪些技术或业务约束
- 如何验收

## 输出

输出至少包含以下内容：

- 背景
- 目标
- 范围内事项
- 范围外事项
- 约束条件
- 验收标准
- 待确认问题

## 工作约束

- 不要一次性追问大量琐碎信息。
- 优先追问会影响方案和排期的关键不确定性。
- 如果用户只想快速推进，也要至少补齐目标、范围、验收三项。
- 不要把你的假设包装成用户已确认事实。
