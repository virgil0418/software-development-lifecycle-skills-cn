# 需求澄清

先把问题定义清楚，再进入设计或编码。

## 目标

输出一份可讨论、可设计、可验收的需求定义，让后续 skill 知道要解决什么、为什么现在做、什么不在本次范围内。

## 澄清步骤

1. 重述用户原始诉求，确认理解一致。
2. 追问业务目标，而不是只盯着表面功能。
3. 区分范围内事项与范围外事项。
4. 明确使用者、受影响方与成功标准。
5. 找出会影响后续方案选择的约束条件。
6. 单列未确认问题，避免把假设误当事实。

## 重点判断

- 这个需求真正要解决的问题是什么。
- 它为什么现在值得做。
- 哪些期待属于误解、扩张或未来版本。
- 如果不做这件事，当前代价是什么。
- 什么结果算成功，什么结果不能接受。

## 输出

输出至少包含以下内容：

- 背景
- 目标
- 用户与影响方
- 范围内事项
- 范围外事项
- 约束条件
- 验收标准
- 待确认问题

## 文档意识

- 当需求范围、非目标、成功标准发生明确变化时，应提醒更新需求类文档或项目 `WORKING` 记录。
- 输出应尽量形成后续 PRD 内容的上半部分，但不要越界替代技术方案。

## 工作约束

- 不要一次性追问大量琐碎信息。
- 优先追问会影响方案和排期的关键不确定性。
- 如果用户只想快速推进，也要至少补齐目标、范围、验收三项。
- 不要把你的假设包装成用户已确认事实。
- 不要在需求未清时直接给出详细技术实现。
