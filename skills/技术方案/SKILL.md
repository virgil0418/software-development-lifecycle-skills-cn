# 技术方案

围绕需求目标形成可评审的技术决策，而不是泛泛而谈的思路说明。

## 目标

给出推荐方案、说明取舍理由、明确实施边界，并把关键限制、风险和未决问题沉淀成后续实现可依赖的技术文档。

## 设计步骤

1. 重述已确认需求与边界。
2. 明确依赖的真实能力、平台约束和外部接口。
3. 给出至少一个可比较的备选方案。
4. 比较复杂度、风险、改动面与长期维护成本。
5. 说明推荐方案及放弃其他方案的原因。
6. 明确模块划分、关键流程、接口、状态流与异常处理。
7. 单列已决事项、未决事项和风险接受边界。

## API 契约要求

- 只要系统存在模块边界、前后端通信、服务间调用或客户端与服务端交互，就应显式定义 API 契约。
- API 契约至少应说明：接口用途、调用方、请求参数、响应结构、错误语义、状态变化、鉴权要求、版本兼容约束。
- 如果当前不写正式 OpenAPI，也应给出最小接口定义，例如字段表、请求/响应示例、错误码说明。
- 对外部 API 要区分已验证事实与文档推断；必要时先实测，再决定实现复杂度。
- 如果接口行为不稳定、依赖第三方协议或存在兼容风险，要显式写入风险与对策。

## 重点判断

- 哪些前提已验证，哪些仍是推断。
- 这个方案最依赖什么能力，最脆弱点在哪里。
- 为什么选它而不是另一个可行方案。
- 哪些问题当前接受，哪些问题必须前置解决。
- 是否需要分阶段落地，而不是一次性做全。
- API 是否已经足够明确到可以支撑开发与测试。

## 输出

输出至少包含以下内容：

- 设计目标
- 真实约束与依赖
- 候选方案
- 方案对比与取舍
- 推荐方案
- 模块划分
- 关键流程
- 数据与接口
- API 契约摘要
- 已决事项
- 未决事项
- 风险与对策
- 分阶段落地建议

## 文档意识

- 输出应尽量形成 RFC 类文档的核心内容。
- 若方案讨论中出现关键实测结论、外部 API 限制或范围收缩，应提醒更新 `WORKING`。
- 若本次方案确定了一条以后可复用的原则，后续可触发 `lessons` 沉淀。

## 工作约束

- 不要把“能做出来”当作“设计合理”。
- 不要跳过替代方案比较。
- 不要忽略性能、稳定性、兼容性、可运维性与安全约束。
- 如果存在明显不确定前提，要显式写出。
- 不要让 API 只停留在“到时候再说”。
- 不要直接越过方案阶段进入任务或代码细节。
