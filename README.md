# 软件开发全生命周期技能库

这是一个面向 Codex 类编码代理的中文 skill 仓库，用来把软件开发从“直接写代码”升级为“先澄清、再设计、再拆解、再实施、再验证、再收尾”的标准化流程。

仓库参考了 `superpowers` 的阶段化思想，但完全使用中文组织内容，并将核心原则从“严格 TDD 唯一正确”调整为更适合 skill 仓库的做法：

- 测试优先
- 验收先行
- 证据收尾
- 修根因而非堆补丁

## 适用场景

- 想为 Codex 建一套中文软件开发技能库
- 希望把需求、设计、编码、测试、评审、交付拆成可复用 skill
- 希望不同任务在不同阶段触发不同工作流，而不是每次都临场发挥
- 希望仓库本身也具备最小可维护性：有文档、有模板、有校验、有回归样例

## 生命周期总览

本仓库当前覆盖以下 9 个技能：

1. 生命周期总控
2. 需求澄清
3. 技术方案
4. 任务拆解
5. 编码实施
6. 测试验证
7. 代码评审
8. 缺陷修复
9. 交付收尾

推荐流转关系：

```text
生命周期总控
  ├─ 需求不清   -> 需求澄清
  ├─ 方案未定   -> 技术方案
  ├─ 任务过大   -> 任务拆解
  ├─ 开始实现   -> 编码实施
  ├─ 需要证明   -> 测试验证
  ├─ 需要把关   -> 代码评审
  ├─ 出现异常   -> 缺陷修复
  └─ 准备结束   -> 交付收尾
```

## 设计原则

### 1. 先定义成功，再开始实现

任何 skill 都不能只告诉代理“做什么”，还要说明“做到什么程度才算完成”。

### 2. 先明确边界，再输出内容

需求不清时不要急着设计，方案未定时不要急着写代码，缺少验收口径时不要声称完成。

### 3. 先交最小闭环，再扩展范围

优先追求最小可验证结果，避免 skill 指令过大、过散、不可落地。

### 4. 优先证据，不靠口头保证

完成、修复、通过、可交付，都需要对应证据：测试结果、复现步骤、影响范围、剩余风险。

### 5. 把流程拆小，把协作说清

每个 skill 只负责一个阶段，阶段之间通过输入输出衔接，避免一个 skill 过度膨胀。

## 使用方式

### 方式一：按阶段手动使用

- 用户需求模糊时，使用“需求澄清”
- 需求清楚但实现方式未定时，使用“技术方案”
- 方案确定但任务太大时，使用“任务拆解”
- 进入实现时，使用“编码实施”
- 需要证明结果时，使用“测试验证”
- 需要质量把关时，使用“代码评审”
- 遇到 bug 时，使用“缺陷修复”
- 准备结束时，使用“交付收尾”

### 方式二：由总控 skill 进行分流

先触发“生命周期总控”，由它判断当前属于哪个阶段，再切换到对应 skill。

## 仓库校验
